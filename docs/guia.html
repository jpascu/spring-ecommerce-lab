Fase 0 — Setup hecho

Objetivo: tener un proyecto Spring Boot arrancable, compilando y con un test de humo.

Conceptos clave

1. Maven y el spring-boot-starter-parent

Maven gestiona dependencias y el ciclo de build. Heredamos de un parent POM que ya define versiones coherentes de todas las librerías de Spring (BOM — Bill of Materials), por lo que no tenemos que indicar versiones manualmente en cada dependencia.

<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.3.5</version>
</parent>

2. Starters

Dependencias "paraguas" que agrupan todo lo necesario para una funcionalidad:

StarterTrae
spring-boot-starter-webSpring MVC, Jackson (JSON), servidor Tomcat embebido
spring-boot-starter-actuatorEndpoints de observabilidad (health, metrics)
spring-boot-starter-testJUnit 5, Mockito, AssertJ, Spring Test

3. @SpringBootApplication

Anotación compuesta. Equivale a tres anotaciones juntas:

  • @Configuration — la clase puede definir beans.
  • @EnableAutoConfiguration — Spring configura automáticamente según lo que hay en el classpath.
  • @ComponentScan — escanea el paquete y subpaquetes buscando componentes.
@SpringBootApplication
public class EcommerceApplication {
  public static void main(String[] args) {
    SpringApplication.run(EcommerceApplication.class, args);
  }
}

4. @RestController y mapeo de rutas

Un @RestController combina @Controller + @ResponseBody: lo que devuelve cada método se serializa directamente a JSON. @RequestMapping define la ruta base y @GetMapping el verbo HTTP.

@RestController
@RequestMapping("/api/ping")
public class PingController {
  @GetMapping
  public Map<String, Object> ping() {
    return Map.of("status", "ok");
  }
}

5. Configuración externalizada (application.yml)

Spring lee la configuración de application.yml/.properties. Aquí definimos puerto, nombre de la app y qué endpoints de Actuator exponer. Esto permite cambiar comportamiento sin recompilar.

6. Test de humo (smoke test)

@SpringBootTest levanta el contexto completo de Spring. Si el contexto arranca sin errores, el test pasa. Es la primera red de seguridad: detecta beans mal configurados.

Relación con tus proyectos de riesgos: esta misma base (starters, autoconfiguración, Actuator, perfiles) es exactamente la que usan los microservicios CBRM. Dominar la Fase 0 te da el esqueleto mental de cualquier servicio Spring Boot corporativo.

Cómo se prueba la Fase 0

Hay tres formas de verificar que el setup funciona, de la más automática a la más manual:

1. Test automatizado (lo ejecuta Maven)

El smoke test EcommerceApplicationTests.contextLoads() arranca el contexto. Para lanzarlo:

# desde la carpeta backend/
mvn test

Si ves BUILD SUCCESS y el banner de Spring Boot en el log, el contexto levanta correctamente.

2. Arrancar la aplicación y llamar a los endpoints

# arranca el servidor en el puerto 8080
mvn spring-boot:run

Con la app levantada, en otra terminal (o en el navegador):

# nuestro endpoint de prueba
curl http://localhost:8080/api/ping
# => {"status":"ok","message":"ecommerce-backend funcionando","timestamp":"..."}

# endpoint de salud de Actuator
curl http://localhost:8080/actuator/health
# => {"status":"UP",...}

3. Compilación completa (empaquetar el jar)

mvn clean package
# genera target/ecommerce-backend-0.0.1-SNAPSHOT.jar
java -jar target/ecommerce-backend-0.0.1-SNAPSHOT.jar
Resumen: mvn test = verificación rápida sin servidor; mvn spring-boot:run + curl = verificación funcional real; mvn package = artefacto desplegable.

Fase 1 — CRUD con arquitectura hexagonal hecho

Implementamos el CRUD de Product sobre JPA + H2 en memoria, pero organizado con arquitectura hexagonal (puertos y adaptadores), DTOs con MapStruct y validación con Bean Validation.

¿Qué es la arquitectura hexagonal?

También llamada Ports & Adapters. La idea central: el dominio (la lógica de negocio) no depende de ningún framework ni tecnología concreta. Todo lo externo (HTTP, base de datos) entra a través de puertos (interfaces) implementados por adaptadores.

  • Puerto de entrada (driving): interfaz que define los casos de uso. La invoca el adaptador web.
  • Puerto de salida (driven): interfaz que el dominio necesita (ej. persistencia). La implementa un adaptador.
  • Regla de oro — dependencias hacia dentro: infraestructura → aplicación → dominio. Nunca al revés.
        HTTP/JSON                          dominio puro                      JPA/H2
   ┌──────────────────┐   in    ┌───────────────────────┐   out   ┌────────────────────┐
   │  ProductController│ ──────▶ │ ProductService (puerto)│ ──────▶ │ ProductRepositoryPort│
   │   (adaptador in)  │         │   ProductServiceImpl   │         │  (puerto out)        │
   └──────────────────┘         └───────────────────────┘         └─────────┬──────────┘
                                                                              │ implementa
                                                                   ┌──────────▼──────────┐
                                                                   │ ProductPersistence  │
                                                                   │ Adapter (JPA)       │
                                                                   └─────────────────────┘

Mapeo de carpetas a la teoría

domain.exception.*Errores de negocio agnósticos al framework

Dos modelos distintos: Product (dominio) y ProductJpaEntity (persistencia) son clases separadas. El ProductPersistenceMapper traduce entre ambas. Así las anotaciones de Hibernate no "infectan" el dominio.

Conceptos clave de la fase

1. Spring Data JPA

El repositorio Spring Data vive en el adaptador. Al extender JpaRepository obtenemos el CRUD sin escribir SQL:

public interface ProductJpaRepository
    extends JpaRepository<ProductJpaEntity, Long> { }

2. Entidad JPA y timestamps automáticos

@Entity mapea la clase a una tabla. @CreationTimestamp / @UpdateTimestamp (Hibernate) rellenan las fechas automáticamente al insertar/actualizar.

3. DTOs + MapStruct (dos mappers)

Usamos record inmutables como DTOs. MapStruct genera los mappers en tiempo de compilación (sin reflexión). En hexagonal hay dos: web↔dominio y dominio↔entidad.

@Mapper(componentModel = "spring")
public interface ProductWebMapper {
  Product toDomain(ProductRequest request);
  ProductResponse toResponse(Product product);
}
MapStruct + Lombok: requieren orden en el annotationProcessorPaths del maven-compiler-plugin: primero lombok, luego lombok-mapstruct-binding, luego mapstruct-processor. Si no, MapStruct no "ve" los getters/setters generados por Lombok.

4. Validación (Bean Validation)

Anotaciones como @NotBlank, @NotNull, @PositiveOrZero en el DTO, activadas con @Valid en el controller. Una petición inválida devuelve 400 automáticamente.

5. Diseño REST del controller

6. Transacciones

@Transactional(readOnly = true) a nivel de clase del servicio; los métodos de escritura lo sobreescriben con @Transactional.

7. Manejo de errores en el borde

La excepción de dominio ProductNotFoundException es pura. El WebExceptionHandler (@RestControllerAdvice) la traduce a 404. En la Fase 2 lo ampliaremos.

Relación con riesgos: la separación puerto/adaptador es ideal para motores de decisión: el "core" de reglas queda aislado y los bureaus externos (Schufa, CCC...) o la persistencia (Dynamo/Postgres) son adaptadores intercambiables y fácilmente mockeables en test.

⬡ Arquitectura hexagonal a fondo

La arquitectura hexagonal (también llamada Ports & Adapters, propuesta por Alistair Cockburn en 2005) es un estilo para organizar el código de forma que la lógica de negocio quede aislada de la tecnología (web, base de datos, mensajería...). Es el concepto vertebral de este proyecto, así que vamos a explicarlo con calma: el problema que resuelve, su vocabulario, cómo se ve en nuestro código y cómo se compara con la arquitectura "clásica" de capas.

1. El problema que resuelve

En una aplicación típica de capas, la lógica de negocio acaba mezclada con detalles técnicos: el servicio importa entidades JPA, conoce HttpServletRequest, llama directamente a un cliente HTTP concreto... El resultado:

  • Cambiar de tecnología (de JPA a Mongo, de REST a Kafka) obliga a tocar la lógica de negocio.
  • Testear el negocio requiere levantar base de datos, web, etc. → tests lentos y frágiles.
  • Las reglas de negocio quedan diluidas entre anotaciones y APIs de frameworks.

La idea central de hexagonal es invertir esa dependencia: que la tecnología dependa del negocio, y nunca al revés.

La "regla de oro" (regla de dependencia): las dependencias siempre apuntan hacia dentro, hacia el dominio. El dominio no sabe que existe Spring, JPA, HTTP ni JSON.

2. La metáfora del hexágono

Imagina la aplicación como una celda con el dominio en el centro. Alrededor hay una frontera con "enchufes" (los puertos). Al exterior conectas adaptadores en esos enchufes. El hexágono no significa "6 lados" literalmente: solo simboliza que hay varios lados por donde conectar tecnologías.

            LADO DRIVING (primario)                 LADO DRIVEN (secundario)
            "quién USA la aplicación"                "qué USA la aplicación"

   adaptadores de ENTRADA          ┌───────────────────────────┐         adaptadores de SALIDA
   ┌─────────────────────┐         │        APPLICATION        │         ┌────────────────────────┐
   │  REST Controller     │        │   (casos de uso / puertos)│         │  JPA Adapter  (H2)     │
   │  Scheduler           │ ──────▶ │  ┌─────────────────────┐ │ ──────▶ │  Lake Publisher        │
   │  Kafka Listener      │ PUERTO  │  │       DOMAIN        │ │ PUERTO  │  Email / Kafka / S3    │
   │  CLI / GraphQL       │   IN    │  │  modelo + reglas    │ │   OUT   │  Cliente API externa   │
   └─────────────────────┘         │  └─────────────────────┘ │         └────────────────────────┘
                                    └───────────────────────────┘
        implementan el adaptador,        define las INTERFACES         el dominio define la interfaz,
        consumen el puerto IN            (puertos) y la lógica          el adaptador la IMPLEMENTA
   ───────────────────────────────────────────────────────────────────────────────────────────────▶
                              las dependencias apuntan SIEMPRE hacia el centro

Truco para no confundir IN/OUT: mira quién manda. Si algo externo llama a la app → es un puerto IN (driving). Si la app llama a algo externo → es un puerto OUT (driven).

4. Driving vs Driven (la clave de la inversión)

La diferencia sutil pero crucial está en quién define la interfaz:

  • Puerto IN: lo define la aplicación y también lo implementa la aplicación (el servicio). El adaptador de entrada solo lo consume.
  • Puerto OUT: lo define la aplicación (lo que necesita), pero lo implementa el adaptador externo. Aquí ocurre la inversión de dependencias: el detalle (JPA) depende de la abstracción (el puerto), no al revés.
// El dominio/aplicación DEFINE lo que necesita (no sabe que detrás hay JPA):
public interface ProductRepositoryPort {        // application/port/out
  Product save(Product product);
}

// La infraestructura lo IMPLEMENTA con una tecnología concreta:
@Component
public class ProductPersistenceAdapter implements ProductRepositoryPort {  // infrastructure
  // ...usa Spring Data JPA, H2, mappers...
}

5. Cómo se mapea en nuestro código

com.lab.ecommerce
├── domain                         # el CENTRO: sin Spring, sin JPA
│   ├── model/Product              # entidad de negocio pura
│   ├── event/ProductChangedEvent  # evento de dominio
│   └── exception/ProductNotFound  # errores de negocio
├── application                    # CASOS DE USO + PUERTOS
│   ├── port/in/ProductService          # puerto IN (lo que ofrecemos)
│   ├── port/out/ProductRepositoryPort  # puerto OUT (lo que necesitamos)
│   ├── port/out/ProductEventPublisherPort
│   └── service/ProductServiceImpl      # implementa el puerto IN
└── infrastructure                 # LOS ADAPTADORES (el "borde")
    ├── adapter/in/web/ProductController        # adaptador primario (REST)
    ├── adapter/in/scheduler/LowStockScheduler  # adaptador primario (tiempo)
    └── adapter/out/persistence/ProductPersistenceAdapter  # adaptador secundario (JPA)
Fíjate que domain y application no importan nada de Spring ni de JPA. Toda la tecnología vive en infrastructure. Esa separación física por paquetes es lo que hace que la regla de dependencia sea fácil de respetar (y de vigilar en revisiones).

6. Flujo de una petición real (crear producto)

Sigamos un POST /api/products de fuera hacia dentro y de vuelta:

1. ProductController (adaptador IN)   recibe JSON → valida @Valid → mapea DTO a dominio
2. llama a ProductService (puerto IN)   ── frontera: a partir de aquí, solo dominio ──
3. ProductServiceImpl (aplicación)     aplica reglas, orquesta el caso de uso
4. llama a ProductRepositoryPort.save (puerto OUT)  ── pide "guárdame esto" ──
5. ProductPersistenceAdapter (adaptador OUT)  traduce a entidad JPA y persiste en H2
6. llama a ProductEventPublisherPort (puerto OUT)  → LakeProductEventPublisher publica el evento
7. el resultado (dominio) sube por la misma cadena; el controller lo mapea a DTO de respuesta (JSON)

Los pasos 1, 5 y 6 son intercambiables: podrías cambiar REST por GraphQL, JPA por Mongo, o el lake por Kafka, sin tocar los pasos 2-4 (el negocio).

7. Hexagonal vs capas clásicas (Controller → Service → Repository)

AspectoCapas clásicasHexagonal
Dirección de dependenciaService depende de Repository (JPA) concretoService depende de un puerto; JPA depende del puerto
Acoplamiento a frameworksEl negocio "ve" JPA, web, etc.El negocio no conoce ninguna tecnología
Testabilidad del negocioSuele necesitar BBDD/contextoTests puros con dobles de los puertos
Sustituir tecnologíaImpacta al negocioSolo cambias el adaptador
Coste inicialMenor (menos clases)Mayor (interfaces + mappers)

Es prima hermana de la Clean Architecture y la Onion Architecture: las tres comparten la misma idea (dominio en el centro, dependencias hacia dentro), solo cambian los nombres.

8. Ventajas y costes (sé honesto)

A favor: negocio aislado y testeable, tecnología intercambiable, límites claros, facilita el trabajo en equipo y la evolución a largo plazo.
Coste: más clases e interfaces, mapeos entre modelos (dominio ↔ JPA ↔ DTO) y una curva de aprendizaje. Para un CRUD trivial puede parecer "demasiado". Merece la pena cuando hay lógica de negocio real y expectativa de vida/cambio (justo el caso de los motores de decisión de riesgos).

9. Preguntas frecuentes

¿El dominio puede usar anotaciones de JPA (@Entity)? En hexagonal estricto, no: la entidad de dominio es pura y la entidad JPA vive en el adaptador (las separamos con un mapper). Algunos equipos las fusionan por pragmatismo, pero pierden parte del aislamiento.

¿Dónde van los DTOs? En el adaptador que los usa (los DTOs web en adapter/in/web/dto). El dominio no conoce DTOs.

¿Un puerto IN por caso de uso o uno por agregado? Ambos son válidos. Nosotros agrupamos el CRUD en ProductService y separamos un caso de uso distinto (StockMonitorUseCase) aplicando el principio de segregación de interfaces.

Relación con tus proyectos de riesgos: un motor de decisión es el caso ideal. El "core" de reglas es el dominio; los bureaus externos (Schufa, CCC), la persistencia y la entrada (REST/cola) son adaptadores. Así las reglas se prueban aisladas y los proveedores externos se cambian o se mockean sin tocar la decisión.

Fase 2 — Buenas prácticas hecho

Tres mejoras transversales sobre el CRUD: manejo global de errores, paginación/ordenación y logging con correlation id (traceId). Todo respetando la arquitectura hexagonal.

1. Manejo global de errores

Centralizamos el tratamiento de excepciones en un @RestControllerAdvice, evitando try/catch repetidos en los controllers y devolviendo siempre el mismo formato de error (ApiError).

ExcepciónHTTPDetalle
ProductNotFoundException404Recurso inexistente
MethodArgumentNotValidException400Validación fallida (lista fieldErrors)
Exception (fallback)500Error no controlado; se loguea, no se filtra al cliente
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ApiError> handleValidation(MethodArgumentNotValidException ex, HttpServletRequest req) {
  List<FieldErrorItem> errors = ex.getBindingResult().getFieldErrors().stream()
      .map(fe -> new FieldErrorItem(fe.getField(), fe.getDefaultMessage()))
      .toList();
  return build(HttpStatus.BAD_REQUEST, "Error de validacion", req, errors);
}

Ejemplo de respuesta 400:

{
  "timestamp": "2026-06-07T09:30:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Error de validacion",
  "path": "/api/products",
  "traceId": "a1b2c3d4",
  "fieldErrors": [ { "field": "name", "message": "el nombre es obligatorio" } ]
}
Alternativa estándar: Spring 6 trae ProblemDetail (RFC 7807). Aquí usamos un record ApiError propio por claridad didáctica, pero el patrón es el mismo.

2. Paginación y ordenación (sin acoplar el dominio)

Para no filtrar Pageable/Page de Spring hacia los puertos, definimos abstracciones propias en la capa de aplicación: PageQuery (entrada) y PageResult<T> (salida). El adaptador JPA las traduce a PageRequest/Page.

// El controller recibe los parámetros y construye un PageQuery
@GetMapping
public PageResult<ProductResponse> findAll(
    @RequestParam(defaultValue="0") int page,
    @RequestParam(defaultValue="20") int size,
    @RequestParam(defaultValue="id") String sort,
    @RequestParam(defaultValue="asc") String direction) {
  return service.findAll(PageQuery.of(page, size, sort, direction)).map(mapper::toResponse);
}

Petición: GET /api/products?page=0&size=5&sort=name&direction=desc. Respuesta:

{ "content": [ ... ], "page": 0, "size": 5, "totalElements": 12, "totalPages": 3 }
PageQuery.of(...) sanea la entrada (page>=0, size acotado a 100, sort por defecto). Nunca confíes en los parámetros que llegan del exterior. PageResult.map() convierte el contenido a DTO conservando los metadatos de paginación.

3. Logging con correlation id (traceId)

Un OncePerRequestFilter (CorrelationIdFilter) asigna un traceId a cada petición y lo mete en el MDC de SLF4J. Todas las líneas de log de esa petición comparten ese id, y lo devolvemos en la cabecera X-Request-Id y en el cuerpo de error.

@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CorrelationIdFilter extends OncePerRequestFilter {
  protected void doFilterInternal(req, res, chain) {
    String traceId = Optional.ofNullable(req.getHeader("X-Request-Id"))
        .orElse(UUID.randomUUID().toString().substring(0, 8));
    MDC.put("traceId", traceId);
    res.setHeader("X-Request-Id", traceId);
    try { chain.doFilter(req, res); } finally { MDC.remove("traceId"); }
  }
}

El patrón de log (en logback-spring.xml) incluye %X{traceId}:

2026-06-07 11:17:31.946 INFO  [a1b2c3d4] c.l.e.a.s.ProductServiceImpl : Producto creado id=4 name='Webcam'
Relación con riesgos: esto es exactamente el mecanismo de un decisionTraceId: propagar un identificador por toda la traza para correlacionar logs entre servicios. Aquí lo aprendes en pequeño.

Cómo se prueba la Fase 2

mvn verify   # ProductControllerIT cubre paginación y formato de error
# manual: enviar tu propio traceId y verlo reflejado en la respuesta
curl -i -H "X-Request-Id: mi-trace-123" http://localhost:8080/api/products
# la respuesta incluye la cabecera X-Request-Id: mi-trace-123

Fase 3 — Patrones de diseño hecho

Implementamos un motor de descuentos que calcula el precio de un producto según el cliente, la cantidad y posibles cupones. Es el caso ideal para combinar cuatro patrones clásicos: Strategy, Template Method, Factory y Builder; y un caso muy cercano a un motor de reglas de riesgos.

El caso de uso

Endpoint POST /api/products/{id}/quote con un cuerpo como:

{ "tier": "VIP", "quantity": 2, "couponCode": "SAVE10" }

Devuelve el presupuesto calculado:

{ "productId": 4, "unitPrice": 100.00, "quantity": 2,
  "subtotal": 200.00, "discount": 20.00, "total": 180.00, "appliedStrategy": "COUPON" }

1. Strategy — algoritmos de descuento intercambiables

Cada regla de descuento es una clase que implementa DiscountStrategy. En lugar de un if/else gigante, añadir un descuento nuevo es añadir una clase (principio abierto/cerrado).

public interface DiscountStrategy {
  boolean supports(PricingContext context);
  BigDecimal calculateDiscount(BigDecimal subtotal, PricingContext context);
  String code();
  default int priority() { return 0; }
}

2. Template Method — esqueleto común del cálculo

AbstractDiscountStrategy fija los pasos comunes en un método final (saneado, tope máximo del 70%, redondeo a 2 decimales) y deja un único hueco para la subclase: doCalculate(). Así ninguna estrategia puede "olvidarse" del tope o del redondeo.

public final BigDecimal calculateDiscount(BigDecimal subtotal, PricingContext ctx) {
  if (subtotal == null || subtotal.signum() <= 0) return ZERO;
  BigDecimal raw = doCalculate(subtotal, ctx);          // paso variable (subclase)
  BigDecimal cap = subtotal.multiply("0.70");          // tope común
  return raw.min(cap).max(ZERO).setScale(2, HALF_UP);   // invariantes garantizados
}
protected abstract BigDecimal doCalculate(BigDecimal subtotal, PricingContext ctx);

3. Factory — elegir la estrategia adecuada

DiscountStrategyFactory recibe todas las estrategias y selecciona la aplicable de mayor prioridad. El caso de uso no sabe nada de estrategias concretas: solo pide "dame la que toca".

public DiscountStrategy select(PricingContext ctx) {
  return strategies.stream()           // ya ordenadas por prioridad desc
      .filter(s -> s.supports(ctx))
      .findFirst()
      .orElseThrow();                  // NoDiscount siempre soporta -> nunca falla
}
Wiring hexagonal: las estrategias y la factoría son dominio puro (sin anotaciones de Spring). Es PricingConfig (en infrastructure/config) quien las declara como @Bean e inyecta List<DiscountStrategy> en la factoría. Spring inyecta automáticamente todos los beans de ese tipo en la lista.
@Bean
public DiscountStrategyFactory discountStrategyFactory(List<DiscountStrategy> strategies) {
  return new DiscountStrategyFactory(strategies);  // Spring rellena la lista con los 4 beans
}

4. Builder — construir el resultado de forma legible

PriceQuote tiene 7 campos relacionados. Un constructor posicional sería ilegible y fácil de equivocar. Lombok @Builder genera el patrón Builder, con una API fluida y autoexplicativa.

return PriceQuote.builder()
    .productId(productId)
    .unitPrice(unitPrice)
    .quantity(ctx.quantity())
    .subtotal(subtotal)
    .discount(discount)
    .total(total)
    .appliedStrategy(strategy.code())
    .build();

Cómo encaja todo (flujo del caso de uso)

PricingService.quote(id, ctx):
  1. productRepository.findById(id)            # puerto OUT (carga el producto)
  2. subtotal = precio x cantidad
  3. strategy = factory.select(ctx)            # FACTORY elige STRATEGY
  4. discount = strategy.calculateDiscount(..) # TEMPLATE METHOD aplica reglas + tope
  5. return PriceQuote.builder()...build()     # BUILDER arma el resultado

Cómo se prueba la Fase 3

# manual

  -H "Content-Type: application/json" \

Relación con riesgos: un motor de decisión es exactamente esto. Cada política (scoring, antifraude, límite) es una Strategy; un Template Method garantiza pasos comunes (auditoría, validación); una Factory elige la política según el producto/canal; y un Builder compone la decisión final. Y al ser dominio puro, todo se testea sin levantar infraestructura.

@ExtendWith(MockitoExtension.class)

5c. Resilience4j (tolerancia a fallos) hecho

Cuando llamamos a un servicio externo (un bureau, una pasarela, un proveedor de precios), ese servicio puede fallar o ir lento. Resilience4j nos da patrones para que su fallo no tumbe nuestra app. Lo aplicamos a un proveedor de tipos de cambio simulado (FlakyExchangeRateAdapter), expuesto en GET /api/exchange-rates/{currency}.

Los patrones

PatrónQué haceAnotación
RetryReintenta los fallos transitorios (un pico puntual)@Retry
Circuit BreakerSi el servicio falla repetidamente, "abre el circuito" y deja de llamarlo un tiempo@CircuitBreaker
FallbackRespuesta alternativa cuando todo falla (valor por defecto, caché...)fallbackMethod
Time LimiterCorta llamadas que tardan demasiado@TimeLimiter
BulkheadLimita llamadas concurrentes para aislar recursos@Bulkhead

El circuit breaker en 3 estados

CLOSED  ──(muchos fallos: >50%)──▶  OPEN  ──(pasados 10s)──▶  HALF_OPEN
   ▲                                                              │
   └──────────────(las llamadas de prueba van bien)──────────────┘

CLOSED    = todo normal, las llamadas pasan al proveedor
OPEN      = el proveedor está "caído": NO se le llama, va directo al fallback
HALF_OPEN = se dejan pasar unas pocas llamadas de prueba para ver si se recuperó

El adaptador protegido

La resiliencia vive en el adaptador de salida (el borde), no en el dominio. El método del fallback debe tener la misma firma más un Throwable:

@Component
class FlakyExchangeRateAdapter implements ExchangeRateProviderPort {

  @Retry(name = "exchangeRate")
  @CircuitBreaker(name = "exchangeRate", fallbackMethod = "fallbackRate")
  public BigDecimal getRate(String currency) {
    // ...llama al proveedor externo; puede lanzar ExternalServiceException
  }

  private BigDecimal fallbackRate(String currency, Throwable t) {
    return BigDecimal.ONE;   // tasa por defecto si el proveedor no responde
  }
}

Y la configuración declarativa en application.yml:

resilience4j:
  retry:
    instances:
      exchangeRate: { max-attempts: 3, wait-duration: 100ms }
  circuitbreaker:
    instances:
      exchangeRate:
        sliding-window-size: 10
        failure-rate-threshold: 50          # % de fallos que abre el circuito
        wait-duration-in-open-state: 10s
Dos detalles importantes: (1) las anotaciones funcionan por AOP (proxies), así que necesitan spring-boot-starter-aop y que la llamada venga de otro bean (no self-invocation). (2) El estado del circuito se publica en /actuator/health (management.health.circuitbreakers.enabled: true).
Relación con riesgos: es justo lo que necesitas al integrar bureaus externos (Schufa, CCC...). Si el bureau cae, el circuit breaker evita saturarlo con reintentos y un fallback aplica una política degradada (p.ej. decisión conservadora) en vez de tumbar el motor entero.
mvn test -Dtest=ExchangeRateResilienceIT
# manual (con la app arrancada):
curl http://localhost:8080/api/exchange-rates/USD    # 1.08 (real)
curl http://localhost:8080/api/exchange-rates/BOOM   # 1 (fallback: el proveedor "falla")

5d. Spring Cache (Caffeine) hecho

La abstracción de cache de Spring permite memorizar el resultado de un método costoso. La aplicamos a los tipos de cambio: cambian poco y obtenerlos implica una llamada externa, así que cacheamos el resultado. La cache envuelve a la resiliencia: un acierto evita por completo la llamada al proveedor (y por tanto el retry/CB).

Cómo funciona

@Service
class ExchangeRateService implements GetExchangeRateUseCase {
  @Cacheable(cacheNames = "exchangeRates", key = "#currency.toUpperCase()")
  public BigDecimal rateFor(String currency) {
    return exchangeRateProvider.getRate(currency);   // solo se ejecuta si NO está en cache
  }
}

Activamos la abstracción con @EnableCaching (en CacheConfig) y usamos Caffeine como proveedor, configurado de forma declarativa:

spring:
  cache:
    type: caffeine
    cache-names: exchangeRates
    caffeine:
      spec: maximumSize=100,expireAfterWrite=5m,recordStats   # TTL 5 min, máx 100, con stats

El trío de anotaciones

AnotaciónQué hace
@CacheableSi hay valor en cache lo devuelve; si no, ejecuta el método y lo guarda
@CachePutEjecuta SIEMPRE el método y actualiza la cache con el resultado
@CacheEvictElimina entradas de la cache (p.ej. al actualizar/borrar el dato)
Cuidado con la coherencia: si cacheas lecturas de una entidad mutable (p.ej. findById de un producto), debes invalidar con @CacheEvict al modificarla, o servirás datos obsoletos. Por eso aquí cacheamos tipos de cambio (cambian despacio) y no el catálogo.
Verificación: ExchangeRateCacheIT mockea el proveedor, pide dos veces la misma divisa y verifica que solo se llama una vez (la segunda es un acierto de cache). Las estadísticas (hits/misses) se publican como métricas Micrometer (cache.gets) gracias a recordStats.
mvn test -Dtest=ExchangeRateCacheIT

5e. Spring Security + JWT hecho

Protegemos la API con autenticación stateless por JWT y autorización por roles. La app actúa como resource server: en cada petición valida el token (no hay sesión en servidor), lo que escala bien y encaja con microservicios.

El flujo

1. POST /api/auth/login {usuario, contraseña}
        └─▶ AuthenticationManager valida credenciales (BCrypt)
            └─▶ JwtTokenService firma un JWT (HS256) con subject + claim "roles"
                └─▶ respuesta { accessToken, tokenType: "Bearer", expiresIn: 3600 }

2. Cada petición protegida envía:  Authorization: Bearer <token>
        └─▶ el resource server valida la firma y la expiración (JwtDecoder)
            └─▶ mapea el claim "roles" a authorities y aplica las reglas de acceso

Reglas de autorización

EndpointAcceso
/api/auth/**, /swagger-ui/**, /v3/api-docs/**, /actuator/**público
GET /api/products/**, /api/exchange-rates/**autenticado (USER o ADMIN)
POST /api/products/*/quoteautenticado
POST/PUT/DELETE /api/productssolo ADMIN

Usuarios de ejemplo (en memoria): user/password (ROLE_USER) y admin/password (ROLE_USER + ROLE_ADMIN).

Piezas clave

// SecurityConfig: cadena de filtros stateless con JWT
http.csrf(disable)                                  // API sin cookies -> CSRF no aplica
    .sessionManagement(STATELESS)
    .authorizeHttpRequests(...)                      // reglas por ruta/método/rol
    .oauth2ResourceServer(o -> o.jwt(...));          // valida el Bearer token

// Codificación/decodificación con secreto HMAC compartido
JwtEncoder = NimbusJwtEncoder(secret);               // firma (login)
JwtDecoder = NimbusJwtDecoder.withSecretKey(secret); // valida (cada request)
Buenas prácticas aplicadas: contraseñas con BCrypt (nunca en claro); el secreto del JWT se inyecta por variable de entorno (JWT_SECRET), no se hardcodea; credenciales inválidas devuelven 401 sin revelar el motivo. En producción los usuarios vendrían de BBDD o de un IdP (Keycloak, Cognito) y se usaría firma asimétrica (RSA/EC) en vez de secreto compartido.
Relación con riesgos: los motores de decisión se exponen como APIs internas que deben autenticar al consumidor y autorizar por scope/rol. El patrón resource server + JWT es exactamente el que usan esos servicios; aquí lo practicas de punta a punta.

Fase 6 — Async & eventos hecho

Desacoplamos los efectos secundarios de la operación principal. Cuando se crea o actualiza un producto, el volcado al data lake ya no se hace en línea (bloqueando la respuesta HTTP): se publica un evento y un listener lo procesa en segundo plano y solo si la transacción confirma.

El flujo de eventos

ProductServiceImpl.create()  (dentro de @Transactional)
   │  guarda en BBDD
   └─▶ eventPublisher.publish(ProductChangedEvent)        // puerto de salida (hexagonal)
         └─▶ ApplicationEventProductPublisher              // adaptador: lo pone en el bus de Spring
               └─▶ applicationEventPublisher.publishEvent(...)
                     · · ·  (la transacción hace COMMIT)  · · ·
                        └─▶ LakeProductEventPublisher.onProductChanged()   // @Async + AFTER_COMMIT
                              vuelca al lago en el hilo "lake-N" (no bloquea la petición)

Las dos garantías

AnotaciónQué aporta
@TransactionalEventListener(phase = AFTER_COMMIT)El listener solo se ejecuta si la transacción confirmó. Si hay rollback, no se vuelca nada incoherente al lago.
@Async("lakeTaskExecutor")Se ejecuta en un hilo del pool propio, así la respuesta HTTP no espera al volcado (latencia y resiliencia).

El pool se define en AsyncConfig con @EnableAsync y un ThreadPoolTaskExecutor (core 2, máx 5, cola 50, hilos lake-*). Un executor propio acota la concurrencia y da nombres útiles en logs.

Por qué un evento y no una llamada directa

El productor (la aplicación) no conoce a los consumidores: publica "ha cambiado un producto" y quien quiera reaccionar se suscribe. Mañana podríamos añadir un segundo listener (invalidar cache, notificar, indexar en buscador) sin tocar el servicio. Se mantiene la hexagonal: la aplicación habla con el puerto ProductEventPublisherPort; el adaptador ApplicationEventProductPublisher traduce eso al bus de Spring.

Detalle clave de @Async: funciona por proxy (AOP), igual que @Transactional. Por eso el listener vive en otro bean (no se auto-invoca) y el método es público. Además, al saltar a otro hilo se pierde el contexto de la petición (Security, MDC del traceId, transacción): si se necesitara, habría que propagarlo explícitamente.
Relación con riesgos: en un motor de decisión la respuesta al cliente no debe esperar a tareas accesorias (auditoría, envío a un lago analítico, notificaciones). Publicar eventos AFTER_COMMIT y procesarlos async mantiene baja la latencia y evita inconsistencias si la operación se revierte.
mvn test -Dtest=ProductEventAsyncIT
# en los logs verás el volcado en un hilo "lake-N", después del commit:
# [LAKE] (async hilo=lake-1) CREATED producto id=42 ...

Ejemplo implementado: enviar cada producto a un data lake

Requisito: "cada producto que se crea o modifica debe enviarse a un lake". En hexagonal esto NO se mete en el controller ni en el repositorio: se modela como un puerto de salida nuevo y un adaptador.

1. Evento de dominio (puro)

// domain/event/ProductChangedEvent.java
public record ProductChangedEvent(Product product, ProductChangeType type, Instant occurredAt) {
  public static ProductChangedEvent of(Product p, ProductChangeType t) {
    return new ProductChangedEvent(p, t, Instant.now());
  }
}

2. Puerto de salida (la aplicación solo conoce esto)

// application/port/out/ProductEventPublisherPort.java
public interface ProductEventPublisherPort {
  void publish(ProductChangedEvent event);
}

3. El caso de uso lo usa sin saber qué hay detrás

// application/service/ProductServiceImpl.java
public Product create(Product product) {
  Product saved = repository.save(product);
  eventPublisher.publish(ProductChangedEvent.of(saved, ProductChangeType.CREATED));
  return saved;
}

4. El adaptador concreto (aquí se simula con log; en real sería S3/Kinesis)

// infrastructure/adapter/out/lake/LakeProductEventPublisher.java
@Component
public class LakeProductEventPublisher implements ProductEventPublisherPort {
  public void publish(ProductChangedEvent event) {
    // real: serializar a JSON/Parquet y PutObject a S3 / putRecord a Kinesis
    log.info("[LAKE] {} producto id={}", event.type(), event.product().getId());
  }
}
Para cambiar de "lake" a Kafka, solo creas otro adaptador que implemente el mismo puerto. El dominio, el servicio y el controller no se tocan. Eso es exactamente lo que aporta hexagonal.

Ejemplo implementado: adaptador de ENTRADA con tarea programada

Un adaptador de entrada (driving) inicia la ejecución de un caso de uso. No tiene por qué ser HTTP: aquí el disparador es el reloj. Implementamos un monitor de stock bajo.

1. Caso de uso (puerto de entrada)

// application/port/in/StockMonitorUseCase.java
public interface StockMonitorUseCase {
  List<Product> findLowStock(int threshold);
}
ISP (Interface Segregation): creamos un puerto aparte en vez de inflar ProductService. Interfaces pequeñas y cohesionadas.

2. El adaptador de entrada (scheduler)

// infrastructure/adapter/in/scheduler/LowStockScheduler.java
@Component
@Profile("dev")
public class LowStockScheduler {
  private final StockMonitorUseCase stockMonitor;

  @Scheduled(initialDelayString = "${app.low-stock.initial-delay-ms:15000}",
             fixedDelayString = "${app.low-stock.interval-ms:60000}")
  public void checkLowStock() {
    stockMonitor.findLowStock(threshold)
        .forEach(p -> log.warn("[STOCK] BAJO id={} stock={}", p.getId(), p.getStock()));
  }
}

Requiere @EnableScheduling en la clase de arranque. Con el perfil dev, al ejecutar verás cada minuto un log [STOCK] .... La lógica (qué es "stock bajo") está en la aplicación; el adaptador solo decide cuándo preguntar y qué hacer con el resultado.

Otros adaptadores de entrada (esquema)

Consumidor de mensajes (Kafka/SQS)

@Component
public class ProductImportListener {
  private final ProductService productService;  // mismo puerto de entrada

  @KafkaListener(topics = "product-imports")
  public void onMessage(ProductImportMessage msg) {
    productService.create(toDomain(msg));   // reutiliza el caso de uso
  }
}

Importador por línea de comandos / fichero

@Component
public class CsvImportRunner implements ApplicationRunner {
  private final ProductService productService;
  public void run(ApplicationArguments args) {
    // lee un CSV y por cada linea: productService.create(...)
  }
}
Patrón común: todos los adaptadores de entrada (REST, scheduler, listener, CLI) acaban llamando al mismo puerto de entrada. Por eso la lógica se escribe una vez y se reutiliza desde cualquier disparador.

¿Cómo elegir entre varios adaptadores del mismo puerto?

Si tienes dos implementaciones (ej. lake y Kafka) Spring necesita saber cuál inyectar:

  • @Profile("lake") / @Profile("kafka"): activas uno u otro por configuración.
  • @Primary: marcas el por defecto.
  • Composite: un adaptador que agrupa varios y publica en todos a la vez.
Cuidado con las transacciones (importante para el lake): publicar dentro de la transacción tiene un riesgo: si la transacción hace rollback después, ya habrás enviado un evento de algo que no se guardó. Patrones recomendados en producción:
  • Eventos tras commit: @TransactionalEventListener(phase = AFTER_COMMIT).
  • Transactional Outbox: guardas el evento en una tabla dentro de la misma transacción y un proceso aparte lo envía al lake. Garantiza consistencia (lo veremos en la Fase 6).
En este ejemplo publicamos de forma síncrona por simplicidad didáctica.

🐳 Testcontainers — Guía en profundidad

Sección destacada por petición expresa. La iremos ampliando con ejemplos ejecutables del proyecto en la Fase 4, pero aquí tienes ya la base conceptual completa.

¿Qué es Testcontainers?

Testcontainers es una librería Java que permite arrancar contenedores Docker reales (bases de datos, brokers de mensajería, etc.) desde tus tests, de forma programática. El contenedor se crea antes del test y se destruye al terminar. Así pruebas contra la tecnología real (PostgreSQL, Kafka, LocalStack...) en lugar de simulaciones.

El problema que resuelve

EnfoqueProblema
BBDD en memoria (H2) para testsH2 no es PostgreSQL: dialectos SQL distintos, tipos como JSONB no existen, funciones propias fallan. "Pasa en test, falla en prod".
BBDD compartida de entornoTests no aislados, datos sucios, dependes de que el entorno esté arriba.
Mocks de todoNo prueban la integración real (SQL, migraciones Flyway, índices).
Testcontainers te da una PostgreSQL real, limpia y efímera por ejecución de tests. Es justo lo que necesitas para los servicios CBRM de Spain (cf-es), que usan PostgreSQL + Flyway + JSONB.

Requisitos

  • Un entorno Docker en marcha (Docker Desktop, Colima, Rancher Desktop, Podman...).
  • Las dependencias de Testcontainers en el pom.xml.

Enfoque A — Clásico con @Testcontainers

Anotas la clase de test, declaras el contenedor como campo static y conectas Spring a la BBDD con @DynamicPropertySource (porque el puerto es aleatorio y solo se conoce en tiempo de ejecución).

@SpringBootTest
@Testcontainers
class ProductRepositoryIT {

  @Container
  static PostgreSQLContainer<?> postgres =
      new PostgreSQLContainer<>("postgres:16-alpine")
          .withDatabaseName("shop")
          .withUsername("test")
          .withPassword("test");

  @DynamicPropertySource
  static void props(DynamicPropertyRegistry registry) {
    registry.add("spring.datasource.url", postgres::getJdbcUrl);
    registry.add("spring.datasource.username", postgres::getUsername);
    registry.add("spring.datasource.password", postgres::getPassword);
  }

  // ... tests que usan el repositorio contra PostgreSQL real
}
static importa: un contenedor static se arranca una vez para toda la clase (rápido). Si no es estático, se crea uno por cada método de test (lento pero más aislado).

Enfoque B — Moderno con @ServiceConnection (Spring Boot 3.1+)

Spring Boot 3.1 introdujo soporte nativo: con @ServiceConnection ya no necesitas @DynamicPropertySource. Spring detecta el contenedor y configura el DataSource automáticamente.

<!-- dependencia extra -->
// org.springframework.boot:spring-boot-testcontainers (scope test)

@SpringBootTest
@Testcontainers
class ProductRepositoryIT {

  @Container
  @ServiceConnection
  static PostgreSQLContainer<?> postgres =
      new PostgreSQLContainer<>("postgres:16-alpine");

  // Spring conecta solo. ¡Sin propiedades manuales!
}
Este es el enfoque recomendado hoy y el que usaremos en la Fase 4.

Patrón "clase base" reutilizable

Para no repetir el contenedor en cada test de integración, se crea una clase base abstracta que todos heredan. Así el contenedor se reutiliza entre clases (Singleton Container pattern).

public abstract class AbstractPostgresIT {
  @Container @ServiceConnection
  static final PostgreSQLContainer<?> POSTGRES =
      new PostgreSQLContainer<>("postgres:16-alpine");
  static { POSTGRES.start(); } // arranque único compartido
}

Uso con @DataJpaTest (slice de persistencia)

Por defecto @DataJpaTest sustituye tu BBDD por una embebida. Hay que desactivar ese reemplazo para que use el contenedor:

@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
class ProductRepositoryJpaIT extends AbstractPostgresIT {
  // inyecta el repositorio y prueba consultas reales
}

Buenas prácticas

  • Fija la versión de la imagen (postgres:16-alpine, no latest) para builds reproducibles.
  • Reutiliza contenedores con contenedor static / clase base: arrancar Docker es lo más lento.
  • Nombra los tests de integración con sufijo IT y sepáralos de los unitarios (failsafe vs surefire).
  • No metas datos de producción; usa datos de prueba generados (EasyRandom encaja muy bien).
  • CI: GitHub Actions ya trae Docker, así que los tests con Testcontainers funcionan sin configuración extra.
  • Considera Ryuk (el contenedor "conserje" que limpia recursos automáticamente al acabar).

Coste / contras

  • Necesitas Docker disponible en local y en CI.
  • Más lentos que un test unitario puro (arranque de contenedor). Mitigado reutilizando contenedores.
